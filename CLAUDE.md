## Project Overview

Studio Architect is a native macOS photography workflow application. It is a **standalone, self-contained product** that manages sessions from SD card import through client delivery and archiving. Every workflow stage (Import, Cull, Edit, Export, Deliver, Archive) is built **natively inside Studio Architect** — it owns its own database and UI shell and has **no runtime dependency on any other app**.

The existing standalone apps (CullSnap, FirstPass, TriPod) are **separate products with no integration into Studio Architect**. During the build-out they serve as low-stakes **feature incubators**: new ideas are tried there first, validated with real users, and then re-implemented natively in Studio Architect. As Studio Architect matures, the standalone apps are wound down. This incubation arrangement is a **transition tactic, not the end state** — once the apps are retired, risky features are trialed inside Studio Architect itself (beta builds / feature flags).

## Tech Stack

- **Language:** Swift
- **UI:** SwiftUI
- **Target:** macOS 15 (Sequoia) minimum
- **Database:** SQLite via GRDB
- **Distribution:** Mac App Store (**sandboxed**)

### Sandbox Constraints (consequences of App Store distribution)

The app is sandboxed, which constrains the full-native pipeline. Build to these:

- **EXIF:** use Apple's **ImageIO** (`CGImageSource`) — **no `exiftool` subprocess** (sandbox forbids spawning arbitrary executables).
- **SD cards / external drives:** access via **user-selected folders + security-scoped bookmarks**, not a full-disk-access entitlement (unavailable to sandboxed apps).
- **Database path:** the sandbox redirects writes into the app **container** (`~/Library/Containers/<bundle-id>/Data/Library/Application Support/StudioArchitect/studio.db`). The "Application Support/StudioArchitect" location below is logically true but physically inside the container.
- **ML culling:** Core ML or a bundled ONNX runtime called **in-process** is allowed (it's a library, not a subprocess).

## Build Commands

The Xcode project is **generated by XcodeGen** from `project.yml` and is **not** committed (`*.xcodeproj` is gitignored). Always generate it first.

```bash
# One-time: install the project generator
brew install xcodegen

# Generate StudioArchitect.xcodeproj from project.yml (run after any project.yml change)
xcodegen generate

# Open in Xcode
open StudioArchitect.xcodeproj

# Build from CLI
xcodebuild -scheme StudioArchitect -configuration Debug -destination 'platform=macOS' build

# Run tests
xcodebuild -scheme StudioArchitect -configuration Debug -destination 'platform=macOS' test
```

## Architecture

### App Structure

```
StudioArchitect/
├── App/                    # App entry point, scene setup
├── Database/               # GRDB setup, migrations, record types
│   ├── AppDatabase.swift   # Database initialization and migrations
│   └── Models/             # GRDB record types (Client, Session, PhotoFile, etc.)
├── Features/               # One folder per major feature/screen
│   ├── Hub/                # Dashboard / session list
│   ├── Session/            # Session detail view
│   ├── Clients/            # Client management
│   └── Team/               # Team member management
└── Shared/                 # Reusable views, extensions, utilities
```

### Database

- Location: `~/Library/Application Support/StudioArchitect/studio.db`
- Migrations live in `AppDatabase.swift` — always add new migrations, never modify existing ones
- All database access goes through GRDB's `DatabaseQueue`

### Standalone Apps (Feature Incubators — No Integration)

Studio Architect does **not** read from, write to, launch, or otherwise depend on the standalone apps. They share **no database and no handoff** — Studio Architect is fully independent. The apps exist as separate products and as a proving ground: a feature is built there first, and once validated it is **re-implemented natively** in Studio Architect (the original code is not reused — it is rebuilt in SwiftUI). The apps are sunset over time.

| App | Stack (standalone) | Native rebuild effort in SA |
|-----|--------------------|------------------------------|
| CullSnap | Electron + TypeScript + React | Full rewrite in SwiftUI (ML via Core ML / in-process ONNX; EXIF via ImageIO, **not** exiftool) |
| FirstPass | SwiftUI (macOS 15) | Low — already native SwiftUI |
| TriPod | Swift + SPM (macOS 13+) | Medium — port to SwiftUI |

## Key Decisions

| Decision | Choice |
|----------|--------|
| Distribution | Mac App Store (sandboxed) |
| Target audience | Hobbyist / side-hustler photographers (simplicity over depth) |
| Database | GRDB / SQLite |
| DB location | `~/Library/Application Support/StudioArchitect/` |
| Edit format | XMP sidecars (industry standard, readable by Lightroom etc.) |
| Checksum | SHA for backup verification |
| App integration | **None** — Studio Architect is fully independent; standalone apps share no data and are sunset over time |
